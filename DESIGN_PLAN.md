# fuckddl — AI 日程管理跨平台应用 设计方案

## Context

开发一款名为 fuckddl 的跨平台（macOS + Android）AI 日程管理应用。核心交互是移动端按住语音输入，AI 自动解析意图并管理日程。日程通过 Git（GitHub 私有仓库）在双端同步。UI 极简果味风格，按键极少。

- **Why**: 现有日程管理工具要么太重（Notion），要么 AI 集成不够自然。期望用语音+AI 实现零摩擦日程管理。
- **Target**: macOS 桌面端 + Android 移动端
- **Tech**: Tauri v2、LangGraph.js、OpenAI 兼容 API、阿里云实时语音转写、Git 同步

## 3. 数据模型

### 3.1 核心数据结构

```typescript
// 日程事件
interface ScheduleEvent {
  id: string;                    // UUID
  title: string;                 // 日程标题
  description?: string;          // 详细描述
  start_time: string;            // ISO 8601 开始时间
  end_time?: string;             // ISO 8601 结束时间（可选）
  all_day: boolean;              // 是否全天事件
  status: 'pending' | 'done' | 'cancelled';
  reminder: Reminder | null;     // 提醒设置
  created_at: string;
  updated_at: string;
  source: 'voice' | 'manual';   // 来源
}

interface Reminder {
  id: string;
  event_id: string;
  remind_at: string;            // ISO 8601 提醒时间
  type: 'notification';         // 提醒类型（未来可扩展）
  triggered: boolean;
}

interface SyncMeta {
  last_sync_at: string;
  last_commit_sha: string;
  repo_url: string;
  branch: string;
}
```

### 3.2 JSON 文件存储布局

```
~/.fuckddl/                     # 应用数据目录
├── events/
│   ├── 2026/
│   │   ├── 04/
│   │   │   ├── event-uuid-1.json
│   │   │   ├── event-uuid-2.json
│   │   │   └── ...
│   │   └── 05/
│   └── 2027/
├── reminders/
│   └── reminder-uuid-1.json
├── sync_meta.json              # 同步元信息
└── config.json                 # 用户配置（API keys 等）
```

按年/月分目录存储每个日程，减少单文件冲突概率，git diff 友好。

### 3.3 Git 同步策略

```
┌──────────────┐     ┌──────────────┐
│   设备 A      │     │   设备 B      │
│  (macOS)     │     │  (Android)   │
└──────┬───────┘     └──────┬───────┘
       │                     │
       │  git push/pull      │
       ▼                     ▼
┌─────────────────────────────────────┐
│         GitHub Private Repo         │
│   (存储 JSON 文件的裸 git 仓库)      │
└─────────────────────────────────────┘
```

同步时机：
- **Pull**: 应用启动时、从后台恢复时、定时（每 5 分钟）
- **Commit + Push**: 每次日程变更后 debounce 2 秒自动提交
- **冲突**: 文件名含 UUID 天然避免大部分冲突；若同文件冲突，以 updated_at 较新者为准

## 4. LangGraph Agent 设计

### 4.1 Agent 状态

```typescript
interface AgentState {
  messages: BaseMessage[];            // 对话历史
  user_input: string;                 // 当前用户输入（文本或 STT 转写结果）
  intent: 'create' | 'query' | 'update' | 'delete' | 'chat' | null;
  extracted_entities: {
    title?: string;
    datetime?: string;                // AI 解析后的时间表达式
    duration?: number;                // 时长（分钟）
    event_id?: string;                // 要操作的日程 ID
    reminder_before?: number;         // 提前多少分钟提醒
  };
  action_result: string;              // 操作结果
  response: string;                   // 给用户的自然语言回复
}
```

### 4.2 Agent 图

```
START
  │
  ▼
[classify_intent]     ← LLM 判断意图（create/query/update/delete/chat）
  │
  ├── create ──→ [extract_create_entities] ──→ [create_schedule] ──┐
  ├── query  ──→ [extract_query_entities]  ──→ [query_schedules] ──┤
  ├── update ──→ [extract_update_entities] ──→ [update_schedule] ──┤
  ├── delete ──→ [extract_delete_entities] ──→ [delete_schedule] ──┤
  └── chat   ──→ [chat_response] ──────────────────────────────────┘
                                                                   │
                                                                   ▼
                                                          [generate_response]
                                                                   │
                                                                   ▼
                                                                  END
```

### 4.4 典型对话示例

```
用户（语音）: "明天下午3点跟产品经理讨论需求，提醒我提前15分钟准备"
  → STT 转写文本
  → Agent: intent=create
  → 提取: { title: "跟产品经理讨论需求", datetime: "2026-04-29T15:00:00", reminder_before: 15 }
  → 创建日程 + 设置提醒
  → 回复: "已创建明天下午3点的「跟产品经理讨论需求」，会提前15分钟提醒你"

用户（语音）: "明天有什么安排"
  → Agent: intent=query
  → 提取: { date: "2026-04-29" }
  → 查询该日所有日程
  → 回复: "明天共有2条日程：下午3点跟产品经理讨论需求，晚上7点团队聚餐"

用户（语音）: "把明天下午的讨论会取消掉"
  → Agent: intent=update
  → 提取: { keyword: "讨论会", date: "2026-04-29", status: "cancelled" }
  → 先 query 匹配到具体 event → update status
  → 回复: "已取消明天下午3点的「跟产品经理讨论需求」"
```

## 6. 设计规范（果味极简主义 · Rams × Jobs 级）

### 6.0 设计宣言

这不是一个"日程管理工具"。这是一个**思维的外置器官**。

用户不是在"操作软件"——用户在想"我明天有什么事"，然后说出来，事情就安排好了。软件的存在感应该趋近于零。每一个多余的按钮、每一段多余的文案、每一个不必要的动画，都是对用户注意力的偷窃。

**核心原则：**
1. **语音是第一公民**。触摸是降级方案，键盘是降级方案。默认状态只有听和说。
2. **今天即一切**。打开应用，看到今天。不需要导航到这个视图——它就是应用本身。
3. **确认而非对话**。AI 的回复不是聊天消息，而是瞬态的确认——像一声"好的"，然后消失。日程的更新才是真正的回复。
4. **触感即信任**。每一个交互都要有物理反馈——弹簧、阻尼、质量感。虚拟的界面要让人感觉像真实的物体。
5. **删除到不能再删**。第一次设计完成后，删掉 50% 的元素，然后再删 50%。剩下的才是 v1。

### 6.1 设计 Token

#### 6.1.1 色彩系统

这不是从任何设计系统抄来的调色板。每一色都经过选择，让它在这个特定场景下唤起正确的情感。

```
核心逻辑：
- 应用是温暖的、私密的（不是冷冰冰的生产力工具）
- 强调色要有紧迫感但不能焦虑
- 灰度要有「体温」——不是中性灰，是微微偏暖的灰
```

| Token | 色值 | 用途 | 为什么是这个颜色 |
|-------|------|------|-----------------|
| `--bg-primary` | `#FAF9F6` | 主背景 | 不是纯白，有 2% 的暖黄，像纸张 |
| `--bg-card` | `#FFFFFF` | 卡片/浮层 | 纯白只在需要层级的表面使用 |
| `--bg-elevated` | `rgba(255,255,255,0.72)` | 毛玻璃面板 | 72% 透明度 + blur，iOS 标准 |
| `--text-primary` | `#1C1C1E` | 主文字 | iOS 系统黑，不是 #000 |
| `--text-secondary` | `#6D6D72` | 次要文字 | 足够对比度但不争夺注意力 |
| `--text-tertiary` | `#AEAEB2` | 占位/辅助 | 存在但几乎不可见 |
| `--accent` | `#FF5A4E` | 强调色/语音球 | 成熟的果实红，有活力但不刺眼 |
| `--accent-soft` | `rgba(255,90,78,0.12)` | 强调色背景 | 像水彩在纸上晕开 |
| `--accent-glow` | `rgba(255,90,78,0.25)` | 录音时光晕 | 像果实被光照透 |
| `--separator` | `rgba(0,0,0,0.06)` | 分割线 | 几乎不可见的边界 |
| `--shadow-card` | `0 2px 8px rgba(0,0,0,0.04)` | 卡片投影 | 只有两层：抬起感+边界感 |
| `--shadow-orb` | `0 4px 20px rgba(255,90,78,0.15)` | 语音球投影 | 让球体"浮"在页面之上 |

**暗色模式适配（v2）：** 背景反转为暖黑（`#1A1A18`），强调色调亮到 `#FF7A70`，投影改为发光效果。

#### 6.1.2 字体系统

```
仅使用系统字体。不引入任何自定义字体。
macOS/iOS: SF Pro Display (title) / SF Pro Text (body)
Android:   Roboto (或系统默认)

理由：系统字体不需要加载，用户潜意识已经熟悉它的形状，
      应用会感觉"原生"而不是 Web 套壳。
```

| Token | 尺寸/行高 | 字重 | 用途 |
|-------|----------|------|------|
| `--text-date-hero` | `40px / 1.1` | 700 | 今天视图的大日期数字 |
| `--text-date-label` | `15px / 1.3` | 400 | 日期下方的星期标签 |
| `--text-event-time` | `15px / 1.4` | 500 | 日程时间 |
| `--text-event-title` | `17px / 1.4` | 400 | 日程标题 |
| `--text-orb-hint` | `13px / 1.3` | 400 | 语音球提示文字 |
| `--text-transcript` | `17px / 1.5` | 400 | 实时转写文字 |
| `--text-confirmation` | `15px / 1.4` | 400 | AI 确认文案 |

字体间距：所有标题使用 `letter-spacing: -0.02em`（iOS 标准），正文使用 `0`。

#### 6.1.3 间距系统

```
基于 4px 原子单位，但实际使用以 8px 为步进。
所有间距必须能被 4 整除。没有例外。
```

| Token | 值 | 使用场景 |
|-------|---|---------|
| `--space-xs` | `4px` | 紧密元素间（时间点和标题之间） |
| `--space-sm` | `8px` | 日程卡片之间 |
| `--space-md` | `16px` | 区块内边距 |
| `--space-lg` | `24px` | 区块之间 |
| `--space-xl` | `32px` | 日期区域与日程列表之间 |
| `--space-xxl` | `48px` | 页面顶部/底部留白 |

#### 6.1.4 圆角系统

```
连续曲线（continuous curve / squircle），不是普通圆角。
iOS 的"果味"很大程度来自这个曲线。
CSS 实现：border-radius 使用百分比或大数值触发连续曲线效果。
```

| Token | 值 | 使用场景 |
|-------|---|---------|
| `--radius-sm` | `8px` | 小元素（标签、指示器） |
| `--radius-md` | `16px` | 日程卡片 |
| `--radius-lg` | `20px` | 浮层、面板 |
| `--radius-full` | `9999px` | 语音球、胶囊按钮 |

#### 6.1.5 动画系统

```
参考 iOS SpringKit 参数。
所有动画使用 CSS cubic-bezier 或 spring 模拟。
关键原则：入场快（0.2s）、出场更快（0.15s）、弹性只在关键交互。
```

| Token | 值 | 使用场景 |
|-------|---|---------|
| `--ease-out` | `0.2s cubic-bezier(0,0,0.58,1)` | 元素出现 |
| `--ease-in` | `0.15s cubic-bezier(0.42,0,1,1)` | 元素消失 |
| `--spring` | `0.5s cubic-bezier(0.34,1.56,0.64,1)` | 语音球按压反馈 |
| `--breathe` | `3s ease-in-out infinite` | 语音球待机呼吸 |
| `--pulse-record` | `0.8s ease-in-out infinite` | 录音中脉冲 |
| `--page-transition` | `0.35s cubic-bezier(0.22,0,0.36,1)` | 页面切换 |

#### 6.1.6 触感反馈映射

```
Android 的 HapticFeedback 常量映射。
每个触感要有明确的情感语义，不是随意震动。
```

| 场景 | Android 常量 | 时机 | 情感 |
|------|-------------|------|------|
| 手指按下语音球 | `KEYBOARD_PRESS` | 手指 touch down | "我听到你了" |
| 开始录音 | `CONFIRM` | 录音阈值触发 | "开始吧" |
| AI 处理完成 | `GESTURE_END` | 日程更新后 | "搞定了" |
| 操作失败 | `REJECT` | 错误状态 | "抱歉，再来一次" |
| 日程被删除 | `LONG_PRESS` + `GESTURE_END` | 删除确认后 | 慎重感 |

### 6.2 移动端——屏幕与状态

移动端只有**两个半屏幕**：
1. **今天**（主屏幕，95% 时间在此）
2. **月历**（从今天视图向左滑动进入）
3. **设置**（从今天视图的极小齿轮图标进入，算半个）

没有底部 Tab Bar。没有汉堡菜单。没有浮动操作按钮。

#### 6.2.1 今天视图——默认状态

```

│  4月28日                          │  ← 40px, weight 700, letter-spacing: -0.02em
│  星期三                           │  ← 15px, weight 400, text-secondary
│                                  │  ← 28px
```

**为什么没有顶部导航栏？**
导航栏是桌面思维的残留。在手机上，用户的拇指在屏幕下半部，目光也在下半部。顶部应该只有内容（日期），不应该有操作。设置入口放在右上角 16px 的小齿轮，是因为设置是低频操作——它应该存在但不抢夺注意力。

**为什么语音球在下方 1/3 位置？**
拇指热区。单手操作时，拇指自然落点就是屏幕下半部。语音球应该精确落在这个区域。直径 72px 是经过计算的：太小会让人犹豫（"我能按准吗？"），太大会像玩具。72px 介于 Apple Watch 的数码表冠和 iPhone 的 home indicator 之间，是一个"不需要思考就能按到"的尺寸。

#### 6.2.3 录音状态（按住时）

这是整个应用最关键的状态。用户在物理上"按住"设备，这一刻的反馈决定了他下次还愿不愿意用语音。

```
┌──────────────────────────────────┐
│                                  │
│  4月28日                          │  ← 背景略微变暗 (overlay: rgba(0,0,0,0.02))
│  星期三                           │     给用户"进入另一个模式"的暗示
│                                  │
│  ┌────────────────────────────┐  │
│  │  15:00 跟产品经理讨论...    │  │  ← 日程列表略微降低透明度
│  └────────────────────────────┘  │     焦点转移到语音交互
│                                  │
│                                  │
│      明天下午三点跟产品经理         │  ← 实时转写文字
│      讨论下周迭代计划             │     17px, text-primary
│                                  │     从语音球上方淡入
│                                  │     文字随识别结果更新
│                                  │     有微弱的闪烁光标效果
│                                  │
│             ●                    │  ← 语音球进入录音态：
│          松开完成                 │     颜色变为 accent 100% 填充
│                                  │     投影增强为 shadow-orb-recording
│                                  │     脉冲动画 (scale 1→1.08, 0.8s)
│                                  │     光晕扩散 30px
│                                  │     提示文字变为"松开完成"
│                                  │

手指按下瞬间：haptic KEYBOARD_PRESS
进入录音阈值：haptic CONFIRM（约 0.3s 后）
录音中：每 2 秒一个极微弱的 haptic TICK（可选）
```

**为什么是"松开完成"而不是"松开发送"？**
"发送"是消息思维——好像你在给 AI 发微信。"完成"是行动思维——你在完成一次表达。后者让用户感觉自己在掌控，前者让用户感觉在请求。两字之差，交互的气质完全不同。

**转写文字的显示策略：**
- 阿里云 STT 返回中间结果时，文字以 0.15s 的淡入动画逐词更新
- 不是打字机效果（太慢，像机器人），而是"呼吸式"——整句替换，旧句 0.1s 淡出，新句 0.15s 淡入
- 如果是静音（用户还没开始说话），语音球脉冲减慢，光晕收敛
- 如果检测到语音结束但没有松手，不自动停止——用户可能只是停顿

#### 6.2.4 处理中状态（松手后）

```

      讨论下周迭代计划              （表示"已收到"）

                                  bounce 动画，每个延迟 0.1s
         正在理解                  ← 13px, text-tertiary

                                  ```

这个状态通常只持续 0.5-1.5 秒。不需要进度条——那会让等待变长。

#### 6.2.5 完成状态（AI 返回后）

```
转写文字和三点动画同时 0.2s 淡出 →

如果创建了日程：
  新日程卡片从语音球位置"生长"出来
  动画：0.4s spring
  - scale: 0.8 → 1
  - opacity: 0 → 1
  - translateY: 8px → 0
  
  同时新卡片在日程列表中从上方插入
  动画：0.35s ease-out
  - 下方已有卡片平滑下移（layoutAnimation）

  语音球恢复待机态，haptic GESTURE_END

如果查询了日程：
  相关日程卡片轻微高亮 1s 后恢复
  高亮：背景变为 accent-soft，0.3s ease

  语音球恢复待机态

如果 AI 需要澄清（比如"你要创建什么日程？"）：
  在语音球上方出现一条半透明文字
  17px, text-secondary
  0.3s 淡入，停留 3s 后自动消失或用户再次按住说话
```

**关键设计决策：AI 的回复不是聊天消息！**

大多数 AI 应用把 agent 回复做成聊天气泡。这在日程管理场景是错的。日程管理不是对话——是行动。用户说"明天下午开会"，期望看到的结果是日历上多了一条日程，而不是一个气泡说"好的我已经帮你创建了日程"。

唯一需要文字反馈的场景：
1. AI 需要用户澄清（歧义消解）
2. 操作失败（错误信息）
3. 查询结果（但应该优先展示日程卡片，文字只是补充）

文字反馈的形式：**Toast 式瞬态提示**，不是聊天气泡。从语音球上方 0.3s 淡入，停留 2-3s 后自动消失。用户不需要"管理对话历史"——日程列表就是对话历史。

#### 6.2.6 错误状态

```
如果 STT 识别失败（没有语音内容）：
  语音球变为危险色 0.3s
  "没有听清，再试一次？"
  13px, text-secondary
  2s 后自动消失
  haptic REJECT

如果网络错误：
  语音球变为琥珀色 0.3s
  "网络不太好，稍后再试"
  2s 后自动消失
  日程数据不受影响（本地优先）

  "出了点问题，再说一遍？"
  同上
```

### 6.6 我在设计中删掉的东西

以下是传统日程应用都有的功能，我刻意删掉了：

1. **底部 Tab Bar（今天/日历/设置）** → 删掉。日历是滑动到达的，设置是低频的。Tab Bar 是导航的懒政。
2. **顶部导航栏** → 删掉。日期本身就是导航。月历中有"← 今天"就够了。
3. **FAB（浮动添加按钮）** → 删掉。语音球就是唯一的操作入口。不需要第二个。
4. **聊天历史列表** → 删掉。日程列表就是对话历史。每一条日程都是你曾经说过的一句话。
5. **手动添加日程的表单** → 删掉（v1）。如果需要手动输入，用文字输入框，不是表单。AI 解析文字和解析语音是一样的。
6. **"确认创建"按钮** → 删掉。AI 解析完直接创建。如果错了，撤销比确认更快。
7. **推送通知的"打开应用"按钮** → 删掉。通知本身就是操作界面。点"知道了"就结束了。
8. **Loading spinner** → 删掉。用三点弹跳代替。Spinner 是"系统在忙"，弹跳是"我在思考"。情感不同。
9. **空状态的插画** → 删掉。插画会过时，文案不会。"说出来，我来记"比任何插画都更有力。
10. **品牌色/Logo 展示** → 删掉。应用的内容就是品牌。日期+日程列表就是最好的"首页"。

### 6.7 设计决策记录

**Q: 为什么语音球是红色而不是蓝色？**
蓝色是科技公司的颜色（Facebook、Twitter、LinkedIn）。红色是人类的颜色——血液、果实、热情。一个帮你管理日程的应用应该是温暖的、私密的、有体温的。红色（准确的说是熟透的珊瑚色）传达"我在乎你的时间"而不是"我在处理你的数据"。

**Q: 为什么没有"添加"按钮？**
因为语音就是添加。打字也是添加。一个独立的"+"按钮分散了核心交互的注意力。当用户习惯语音后，手动添加是降级方案——降级方案不应该和主方案争夺同一块视觉空间。

**Q: 为什么 AI 回复不是聊天界面？**
聊天界面适合"探索型"交互——你不知道 AI 能做什么，你在试探。日程管理是"意图型"交互——你明确知道你要什么，你只需要说出来。聊天界面在这种场景下是多余的一层抽象。

**Q: 语音球为什么要呼吸动画？**
静态的圆是死的。呼吸的圆是活的。一个微微起伏的圆给人的潜意识暗示是："我在听，我准备好了"。这不只是装饰——这是 affordance。

**Q: 月历为什么是滑动而不是 Tab？**
Tab 是离散的——你在两个不相关的空间之间跳转。滑动是连续的——月历是今天的自然延伸。"明天有什么"和"下周三有什么"是同一类问题，不应该被 UI 强行分开。

## 9. 关键风险与应对

| 风险 | 应对 |
|------|------|
| Android 上 Git 操作不可靠 | 使用 git2 纯 Rust 实现，不依赖系统 git |
| Git 冲突导致数据丢失 | UUID 文件名 + updated_at 冲突策略 + 操作前自动 pull |
| 阿里云 STT 延时影响体验 | 实时模式 WebSocket 流式返回，边说话边显示中间结果 |
| Android 后台进程被杀死 | 提醒使用系统 AlarmManager 调度，不依赖应用进程 |
| LangGraph.js 在移动端 WebView 性能 | Agent 图保持轻量（< 5 个节点），LLM 推理走云端 |

## 10. 验证方式

- **数据层**: 单元测试 Rust JSON store CRUD，验证读写正确性
- **同步**: 手动在 A 设备创建日程 → commit push → B 设备 pull 验证
- **Agent**: 用一组标准测试句子验证意图分类和实体抽取准确率
- **语音**: 端到端测试录音→转写→Agent→日程创建完整链路
- **UI**: macOS 开发环境热更新验证，Android 模拟器/真机验证
- **跨平台**: 两端分别操作后确认 git 同步数据一致
